## 0. Before you start — confirm, don't assume

1. **Hosting environment**: this spec assumes Vercel serverless (functions with execution time limits, no persistent filesystem outside `/tmp`, `/public` baked at build time). If the app is actually running on a persistent server, skip the Blob storage steps in §3 and §7 and write generated assets to local disk instead, consistent with the existing `/public/generated-dna/` pattern. **Confirm this before Phase 2.**
2. **Env vars needed** (add to `.env`):
   - `GOOGLE_API_KEY` — Nanobana access (already provisioned per El-Roy)
   - `BLOB_READ_WRITE_TOKEN` — Vercel Blob (if hosting assumption in §0.1 holds)
   - `QSTASH_TOKEN`, `QSTASH_CURRENT_SIGNING_KEY`, `QSTASH_NEXT_SIGNING_KEY` — Upstash QStash
   - Existing `DEEPSEEK_API_KEY` and Neon connection string are already configured — reuse them, don't create new clients.
3. **New npm packages**: `@upstash/qstash`, `@vercel/blob`, `sharp`, `qrcode`. All lightweight, no heavy infra.

## 1. Database migration

Add to `prisma/schema.prisma`:

```prisma
model BrandVisualIdentity {
  id                 Int                  @id @default(autoincrement())
  waitlistId         Int                  @unique
  waitlist           WaitlistBrandDNA     @relation(fields: [waitlistId], references: [id])

  status             GuidelineStatus      @default(NOT_STARTED)
  archetypeCluster   String?

  brandColorHex      String?
  typographyJson     Json?

  logoUrl            String?
  imageryUrls        String[]             // 3 Values images
  mockupUrls         Json?                // { businessCards: [...], appIcon, socialFrames: [...], tier2: {...} }

  failureReason      String?
  emailSentAt        DateTime?
  createdAt          DateTime             @default(now())
  updatedAt          DateTime             @updatedAt
}

model MockupTemplate {
  id                 Int      @id @default(autoincrement())
  templateId         String   @unique
  category           String   // "apparel_worn" | "tote_bag" | "billboard" | "door_hanger" | "keychain" | "busstop"
  baseImageUrl       String
  garmentMaskUrl      String?
  tintEnabled        Boolean  @default(false)
  logoZoneJson       Json     // { xPct, yPct, widthPct, rotationDeg }
  active             Boolean  @default(true)
  createdAt          DateTime @default(now())
}

enum GuidelineStatus {
  NOT_STARTED
  QUEUED
  GENERATING
  COMPLETE
  FAILED
}
```

Update `WaitlistBrandDNA` to add the back-relation:

```prisma
model WaitlistBrandDNA {
  // ...existing fields...
  visualIdentity BrandVisualIdentity?
}
```

Run migration: `npx prisma migrate dev --name add_brand_visual_identity`

## 2. Phase 1 — Content template (no AI, ship first)

### 2.1 Content mapping function

Create `lib/visualIdentity/mapDnaToTemplate.ts`. This reshapes existing `dnaJson` into the page's content slots — **do not call DeepSeek here**, this is a pure reshape of content already generated by the discovery questionnaire flow.

```ts
interface VisualIdentityContent {
  brandName: string;
  aboutUs: string;        // from dnaJson origin/offer synthesis
  mission: string;        // from dnaJson future/philosophy answers
  vision: string;
  values: { label: string; description: string }[]; // exactly 3, from dnaJson personality/character section
  slogan: string;
  toneWords: string[];    // 3, ranked, from dnaJson voice/tone output
}

function mapDnaToTemplate(dnaJson: BrandDNA): VisualIdentityContent { /* ... */ }
```

If any of these fields don't cleanly exist in the current `dnaJson` shape (double-check against the actual questionnaire output before assuming), that's a signal to either add them to the DeepSeek synthesis prompt in the existing DNA-generation route, or derive them with a single lightweight DeepSeek call here — but check the existing shape first, this should mostly be free.

### 2.2 Page route

`app/waitlist/[id]/visual-identity/page.tsx` — server component, fetches `BrandVisualIdentity` + `WaitlistBrandDNA`, renders sections in this fixed order:

1. Cover (brand name + logo, or placeholder state if `status !== COMPLETE`)
2. Logo (full color / on-dark / mono treatments — derive the on-dark and mono versions from the single generated PNG via `sharp` invert/threshold, don't generate 3 separate images)
3. Colors (single brand color swatch + hex/RGB/HSL, styled like the Getly reference)
4. Fonts (heading/body pairing preview)
5. About Us (text + one Values image)
6. Mission/Vision
7. Values (3 cards, each with its generated image + label overlay)
8. Slogan/Voice (tagline + tone indicators)
9. Mockups gallery (Phase 3/4 output, empty/hidden until those phases ship)

Style the page live using the brand's color/fonts as CSS custom properties (`--brand-color`, `--font-heading`, `--font-body`) set inline on the root container.

### 2.3 Status states without AI yet

For Phase 1, `BrandVisualIdentity.status` can go straight to `COMPLETE` once the content mapping succeeds — no async pipeline needed until Phase 2 introduces Nanobana calls.

**Phase 1 done when**: a user can click through from DNA approval and see a fully content-populated (no generated logo/color/imagery yet — use sensible defaults) guideline page.

## 3. Phase 2 — Logo, color, Values imagery, async pipeline, email

### 3.1 Trigger

Add a CTA on the DNA-approval screen: **"Generate your Visual Identity →"**. On click:

```ts
// POST /api/brand-dna/[id]/visual-identity
// 1. Create BrandVisualIdentity row, status: QUEUED
// 2. Publish QStash message with { waitlistId }
// 3. Return 202 immediately, redirect user to the progress screen
```

Rate-limit: reject if a `BrandVisualIdentity` row already exists for this `waitlistId` (one generation per brand).

### 3.2 QStash worker

`app/api/brand-dna/process-guideline/route.ts` — receives the QStash webhook, verified via QStash signing keys.

```
1. Set status: GENERATING
2. Classify archetype cluster from dnaJson (see §3.3)
3. Generate + validate brand color (see §3.4)
4. Look up typography pair (deterministic, see §3.5)
5. Generate logo mark via Nanobana (see §3.6)
6. Generate 3 Values images via Nanobana (see §3.7)
7. Upload logo + images to Blob storage
8. status: COMPLETE
9. Trigger email send (see §3.8)
```

Wrap each step in try/catch; on any failure set `status: FAILED` and `failureReason`, but still attempt to serve whatever succeeded (don't block the whole page on one failed asset — fall back to a solid-color placeholder for that slot).

### 3.3 Archetype classification

Single DeepSeek call, structured JSON output only:

```
System: "Classify this brand into exactly one archetype cluster from this list: agro_earth, corporate_trust, bold_consumer, premium_luxury, tech_minimal. Respond with only the cluster key, nothing else."
User: [relevant dnaJson fields — industry, personality words, tone]
```

### 3.4 Brand color generation

```
1. DeepSeek: given the archetype cluster + brand personality words, generate 1 candidate hex color within that cluster's hue/saturation range (pass the cluster's constraints in the prompt — see palette table in the proposal doc §3.4/Rev.3 §... for the 5 cluster definitions).
2. In code: check WCAG contrast ratio of the candidate against both #FFFFFF and #000000.
3. If contrast fails on both, nudge lightness in code (not another AI call) until it passes, or regenerate once.
```

### 3.5 Typography lookup

Deterministic object, no AI:

```ts
const TYPE_PAIRS: Record<ArchetypeCluster, { heading: string; body: string }> = {
  agro_earth: { heading: "Clash Display", body: "DM Sans" },
  corporate_trust: { heading: "Plus Jakarta Sans", body: "Inter" },
  bold_consumer: { heading: "Bricolage Grotesque", body: "General Sans" },
  premium_luxury: { heading: "Fraunces", body: "Inter" },
  tech_minimal: { heading: "Space Grotesk", body: "Inter" },
};
```

(Adjust actual font choices as needed — the point is a fixed lookup table, no generation.)

### 3.6 Logo generation prompt

One Nanobana call:

```json
{
  "subject": "A simple, flat, geometric icon mark representing [brand core concept, 1-2 words from dnaJson archetype/offer]",
  "style": "Minimalist vector-style flat icon, single accent color on white or transparent background, bold clean geometric shapes, centered composition, works at small sizes",
  "color": "[generated brand color hex]",
  "negative_prompts": [
    "no photorealism",
    "no gradients",
    "no text or lettering",
    "no 3D rendering",
    "no drop shadows"
  ],
  "aspect_ratio": "1:1",
  "output_notes": "Should read clearly as a simple icon/badge, similar in spirit to a flat app icon glyph"
}
```

Post-process: crop/pad to a clean square, ensure transparent background if the model doesn't provide one (basic background removal via a solid-color flood-fill from the corners, or ask Nanobana explicitly for a transparent/white background and threshold it in code).

### 3.7 Values imagery prompts

Loop 3x, one per value, using the archetype cluster's photography style (see proposal doc Rev.2 §4.4 for the cluster-to-style mapping) plus the specific value's label/description as the subject anchor. Same JSON prompt schema as §3.6 but for photography rather than icon style — `style: "editorial contextual photography..."` etc.

### 3.8 Email

Use the existing email provider already wired up in the DNA-approval flow (check what's configured — Resend/SendGrid per the original draft). Template: hosted image URLs (not inline attachments), brand color as an accent in the email chrome, single CTA button back to `/waitlist/[id]/visual-identity`.

### 3.9 Frontend polling

Progress screen polls `GET /api/brand-dna/[id]/visual-identity/status` every 2–3s while `status` is `QUEUED` or `GENERATING`, redirects to the full page on `COMPLETE`. Keep the loading copy warm and specific ("Sketching your mark...", "Finding your colors...") rather than a generic spinner — this is a delight moment, not a wait screen.

**Phase 2 done when**: clicking the CTA produces a real logo, brand color, and 3 Values images within ~30–60s, and an email arrives.

## 4. Phase 3 — Tier 1 code mockups (no AI, high visual payoff)

All components take the same base props: `{ logoUrl, brandName, brandColor, fontPair, tagline? }`.

### 4.1 Business cards — 5 layout variants

`components/mockups/BusinessCard/[Variant1-5].tsx`. Match the layout variety from the uBrand reference: logo-top-left-info-right, logo-top-right-with-wave-bg, color-block-band-bottom, logo-top-right-minimal, vertical-line-separator. Each renders to a fixed-aspect canvas (standard business card ratio, 3.5:2) and exports to PNG via server-side rendering (e.g. `@vercel/og` or a headless render step) for the emailed/downloadable version.

**Nice touch**: generate a real QR code (via the `qrcode` package, zero cost) pointing to the brand's own `/waitlist/[id]/visual-identity` page instead of a placeholder QR — makes the mockup feel functional rather than decorative.

### 4.2 App icon

Single component: rounded-square frame, brand color background, logo centered at ~60% of frame width. Export at standard app icon sizes if useful (512x512 minimum).

### 4.3 Social post frames — 3 variants

Matching the "Trending Topic" / "Team Introduction" / "About Us" pattern from the reference: diagonal triangle or color-block background (SVG, `fill: var(--brand-color)`), a text slot pulling from `dnaJson` (a value, the tagline, or a mission line), and an optional image slot that reuses one of the 3 Values images from Phase 2 rather than generating anything new.

**Phase 3 done when**: the guideline page's "Mockups" section shows business cards, an app icon, and 2-3 social post templates, all correctly branded, all rendered with zero additional Nanobana calls.

## 5. Phase 4 — Tier 2 photographic template library (one-time build)

This is a **one-time setup task**, not part of the per-brand pipeline. Build a small admin/seed script, not a user-facing flow.

### 5.1 Template list (MVP set — 5 templates)

Generate each once via Nanobana using the JSON prompt schema below, review manually, then insert into `MockupTemplate`:

1. `apparel_tshirt_model_front_a` — category: `apparel_worn`, `tintEnabled: true`
2. `tote_bag_front_a` — category: `tote_bag`, `tintEnabled: true`
3. `keychain_front_a` — category: `keychain`, `tintEnabled: true`
4. `billboard_urban_a` — category: `billboard`, `tintEnabled: false`
5. `door_hanger_wood_a` — category: `door_hanger`, `tintEnabled: false`

(Expansion set for later, not MVP: `apparel_polo_model_a`, `apparel_pair_friends_a`, `busstop_ad_a`.)

### 5.2 Prompt schema (use for every template)

```json
{
  "template_id": "<unique id>",
  "category": "<category>",
  "subject": "<description of the blank/neutral scene>",
  "style": "<photographic style, lighting, framing>",
  "logo_placement_zone": {
    "description": "<where the logo goes, in plain language>",
    "approx_size_pct_of_frame_width": <number>,
    "vertical_position": "<description>"
  },
  "garment_recolor_target": "<what should be tinted, or null>",
  "negative_prompts": [
    "no existing logos, text, or branding",
    "<other constraints specific to this category>"
  ],
  "aspect_ratio": "<ratio>",
  "output_notes": "<any additional guidance, e.g. keep base neutral for tinting>"
}
```

Concrete examples for `apparel_tshirt_model_front_a`, `billboard_urban_a`, and `tote_bag_front_a` are in the proposal doc (Rev. 3, §5) — use those verbatim as starting prompts, iterate based on actual Nanobana output quality.

### 5.3 Manual step: logo zone + mask

After generating each base image:

1. Manually determine `logoZoneJson` (`xPct`, `yPct`, `widthPct`, `rotationDeg`) by inspecting the image — this is a five-minute task per template, not worth automating for a library this small.
2. For `tintEnabled: true` categories, crop/export a simple mask (white = tintable region, black = everything else) as `garmentMaskUrl`. A rough manual mask is fine — doesn't need pixel-perfect edges.

### 5.4 QA gate

**Do not flip any template to `active: true` without a manual visual check.** These are reused across every brand indefinitely — a bad billboard template is a bad billboard template for every single customer until someone notices and fixes it.

### 5.5 Compositing pipeline (per brand, runs in the Phase 2 worker once Phase 4 ships)

```ts
async function compositeMockup(template: MockupTemplate, brand: { logoUrl: string; brandColorHex: string }) {
  const base = sharp(await fetchAsBuffer(template.baseImageUrl));
  const meta = await base.metadata();

  let working = base.clone();
  if (template.tintEnabled) {
    const rgb = hexToRgb(brand.brandColorHex);
    working = working.tint(rgb); // sharp's built-in tint, applied to the neutral base
  }

  const zone = template.logoZoneJson as { xPct: number; yPct: number; widthPct: number };
  const logoWidth = Math.round(meta.width! * (zone.widthPct / 100));
  const logoBuffer = await sharp(await fetchAsBuffer(brand.logoUrl))
    .resize(logoWidth)
    .toBuffer();

  const composited = await working
    .composite([{
      input: logoBuffer,
      left: Math.round(meta.width! * (zone.xPct / 100)),
      top: Math.round(meta.height! * (zone.yPct / 100)),
    }])
    .png()
    .toBuffer();

  return uploadToBlob(composited, `mockups/${brand.waitlistId}/${template.templateId}.png`);
}
```

Note: `.tint()` applies uniformly to the image, which is fine for these base photos since they're shot specifically to be neutral/tintable — if a template later needs region-limited tinting (tint only the garment, not the model's skin/background), swap this for a mask-based composite using `garmentMaskUrl` as a blend mask instead. Start with the simpler uniform tint and only add masked tinting if the uniform version looks wrong on real output.

### 5.6 Selecting a variant per brand

Deterministic, not random, so re-generation (if ever needed) is stable:

```ts
function pickVariant(category: string, waitlistId: number, templates: MockupTemplate[]) {
  const candidates = templates.filter(t => t.category === category && t.active);
  const index = waitlistId % candidates.length;
  return candidates[index];
}
```
